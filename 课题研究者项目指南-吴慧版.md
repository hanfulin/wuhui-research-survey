## 0.5 认知负担地图：哪些要懂、哪些只要说、哪些非你不可

### 🧠 A类：只要"懂"就好（概念，不用自己写）

- 前端 vs 后端、NoSQL 的集合/文档
- `onSnapshot` 订阅（数据变动自动刷新）
- 白名单安全规则、静态 vs 动态资源该放哪个"家"
- GitHub 版本控制的基本逻辑（每次修改都有记录）

### 💬 B类：只要"说"就好（AI 完全代劳）

| 范围 | AI 帮你做的事 |
|------|-------------|
| GitHub | `git init`、commit、push、开 Pages、回退旧版本 |
| Firebase | 建项目、建数据库、写规则、部署、收问卷数据 |
| Obsidian | 建同名文件夹、更新研究日志、写踩坑记录 |
| 问卷网页 | 写所有 HTML/CSS/JS、问卷逻辑、统计展示 |

第5节"句型模板库"就是这类指令的**复制粘贴模板**——照抄改关键字就好。

### 🫳 C类：只有"你"能做（AI 无法代劳）

1. **物理动作**：在本地开文件夹、登录 Firebase/GitHub、打印问卷二维码
2. **决策判断**：问卷题目设计、访谈提纲确认、哪些数据要收集
3. **责任把关**：上线前的受访者隐私确认、按下发布让受访者看到的那一刻

## 1. 三个"家"的角色定位

### 核心洞察：每个课题有三种不同的数据

| 数据类型 | 特性 | 该住哪 |
|---------|------|-------|
| **网页/问卷代码** | 你（Claude）写的、不会自己变 | **GitHub** |
| **受访者提交的问卷数据** | 使用者互动产生、即时在变 | **Firebase** |
| **思考与研究日志** | 你动脑产生、给未来的你看 | **Obsidian** |

---

### 🐙 GitHub 的角色：网页与代码的"家"

**一句话**：你做的问卷网页、分析工具，放在这里让研究团队访问。

#### 核心概念

- **版本控制**：每次 `commit` = 一个存档点，可以随时回到过去某个版本
- **静态资源**：HTML/CSS/JS 上传后不会自己改变
- **公开发布（GitHub Pages）**：让问卷网页有公开网址，免费，不用租服务器

#### 你的课题里 GitHub 存什么

- `survey.html`（问卷提交页，发二维码给受访者扫）
- `results.html`（统计展示页，团队内部查看）
- 报告草稿（`.md` 格式的文字文件）
- `.gitignore`（排除隐私数据）

#### 常见错误

- ❌ 把受访者姓名写进 HTML 里推上去 → 公开仓库所有人都看得到
- ❌ 改完文件忘了 commit → 电脑出问题就丢了
- ❌ commit 信息都写"更新" → 三个月后自己看不懂

---

### 🔥 Firebase 的角色：问卷数据的"家"

**一句话**：受访者填写问卷后，数据都住在这里。

#### 核心概念

- **NoSQL 文档数据库（Firestore）**：不像 Excel 每行字段都要一样，每条"文档"可以有不同字段
- **即时同步（onSnapshot）**：数据一变动，所有打开的页面自动更新，不用刷新
- **安全规则**：决定谁能读、谁能写；默认应该是"白名单"

#### 你的课题里 Firebase 存什么

| 集合名（建议） | 存什么 | 谁能写 |
|-------------|--------|--------|
| `survey_responses` | 问卷回答（科研人员、管理人员、财务人员） | 公开可写（受访者） |
| `interview_notes` | 访谈关键记录（去标识化） | 仅课题团队 |

#### 免费（Spark）方案对你够不够用

你的受访者规模预计是医院内部相关人员，远低于每天读取50,000次的免费上限，**完全够用，不用担心费用**。

#### 常见错误

- ❌ 把受访者真实姓名存进去 → 数据会经过 AI 分析，存在隐私风险
- ❌ 安全规则选"测试模式"30天后过期 → 应该直接写白名单
- ❌ 安全规则全开放 → 全世界都能改你的数据

---

### 📘 Obsidian 的角色：研究思考与进展的"家"

**一句话**：给人看的研究日志，不是给程序读的。

#### 核心概念

- **文档即代码（Docs as Code）**：全部用 Markdown 写，Claude 最容易看懂
- **单一数据源（SSOT）**：进度只写一个地方，避免"写了多份、久了不一致"
- **双向链接**：`[[笔记名]]` 相关笔记互相串起来

#### 你的课题里 Obsidian 存什么

| 文件 | 内容 |
|-----|------|
| `研究进展日志.md` | 每阶段完成情况、当前进度 |
| `踩坑记录.md` | 遇到问题、怎么解决的 |
| `决策日志.md` | 为什么这样设计问卷、为什么选这个分析方法 |
| `文献笔记/` | 每篇参考文献的摘要与启发 |
| `CLAUDE.md` | 课题说明书（Claude 每次开始前会读） |

#### 常见错误

- ❌ 进度写在三个地方（Obsidian / Word / 微信）→ 双写漂移
- ❌ 踩坑当下没记，下次再踩 → 浪费时间
- ❌ 把受访者姓名放 Obsidian → Obsidian 同步到云端存在隐私风险

---

### 三个"家"怎么合作？以"科研经费包干制问卷"为例

| 步骤 | 动作 | 动到哪个"家" |
|------|------|-----------|
| 1 | 设计问卷题目 | **Obsidian** — 开新笔记写问卷设计思路 |
| 2 | 对 Claude 说"帮我建项目" | **GitHub** — 建仓库、push 初始提交 |
| 3 | 写问卷提交页 `survey.html` | **GitHub**（文件）+ **Obsidian**（记录设计决策） |
| 4 | 建 Firebase 数据库收数据 | **Firebase** — 建集合 + 安全规则 |
| 5 | 写统计展示页 `results.html` | **GitHub**（代码）+ **Firebase**（数据来源） |
| 6 | push 到 GitHub Pages | **GitHub** — 部署上线 |
| 7 | 受访者扫二维码填写 | **Firebase** — 收数据 |
| 8 | 团队查看统计结果 | **Firebase → GitHub 上的网页** |
| 9 | 对 Claude 说"帮我分析问卷数据" | **Firebase** — 分析 |
| 10 | 记录今天的进度 | **Obsidian** — 更新研究进展日志 |

---

### 核心原则：每份数据只有一个"家"

| 数据类型 | ✅ 正确的家 | ❌ 绝对不该去的地方 |
|------|---------|-------------------|
| 问卷网页 HTML/CSS/JS | GitHub | Firebase、Obsidian |
| 受访者提交的问卷数据 | Firebase | GitHub、Obsidian |
| 研究进展、决策日志 | Obsidian | 程序注释、commit 信息 |
| 受访者真实姓名 | 不存（去标识化） | 任何地方 |
| Firebase 公开配置（`firebaseConfig`） | 直接写进 HTML（这是公开信息） | — |

**口诀：静态去 GitHub、数据去 Firebase、想法去 Obsidian。**

## 🚨 3.5 暂停一下：这套做法什么时候不该用？

### ✅ 这套做法适合什么

- 课题团队内部工具（≤10人管理，受访者匿名）
- 研究期限内的工具（8个月）
- 数据可去标识化（只用部门/职称/年限区间）
- 吴慧本人是主要维护者，出问题能立刻找 Claude 修

### ❌ 这套做法不适合什么

| 场景 | 为什么不适合 | 建议做法 |
|------|------------|---------|
| 存储受访者真实姓名+工号 | 涉及医院员工个人信息，有隐私合规风险 | 匿名问卷+分类标签 |
| 替代医院正式信息系统 | 需要稽核、安全认证 | 用医院既有系统 |
| 全院1000+人同时填写 | 免费额度可能不足 | 升级Firebase方案或换工具 |
| 存储敏感财务数据（具体金额/合同） | 数据安全要求高 | 在医院内网处理 |

## 6. 常见问题（提前知道）

### GitHub 篇

| 问题 | 解法 |
|----|------|
| 私有仓库不能用 GitHub Pages | 改公开仓库，或付费升级 |
| 中文文件夹名在 GitHub URL 很难看 | 仓库用英文名（`wuhui-research-survey`） |
| commit 信息都写"更新" | 写清楚"做了什么+为什么"，三个月后的你会感谢 |

### Firebase 篇

| 问题 | 解法 |
|----|------|
| 项目名被自动加后缀（如 `my-tool-a3b9c`） | 建好后到 Project Settings 确认真正的 Project ID |
| 安全规则选"测试模式"30天过期 | 一律选"正式版模式"+立刻写白名单 |
| `firebaseConfig` 能公开吗？ | **可以公开**，apiKey 和 projectId 设计给前端用 |

### Obsidian 篇

| 问题 | 解法 |
|----|------|
| 进度写在三个地方（Obsidian/Word/微信） | 遵守单一数据源原则，进度只写 `研究进展日志.md` |
| 不同电脑之间不同步 | 把 Obsidian vault 建在同步盘里（你已经在 SynologyDrive 了） |

### 隐私/数据安全篇

| 问题 | 解法 |
|----|------|
| 受访者真名进了 AI | 问卷设计为匿名，只收部门/职称/年限 |
| Firebase 全开放任何人都能改数据 | 立刻改白名单：只开 `survey_responses_2026` 集合 |
| `.env` 不小心 commit 上去 | `.gitignore` 一定要把 `.env`、`*.key` 列进去 |
